## Execution Cost Map
> Architectural awareness, NOT an optimization to-do list. Not every region here is a problem; some are inherent and fine.

Scope: backend read pipeline for `GET /contracts/` (paginated list with filter/sort/search). Reasoning is static-only — from call-site count, loop nesting, per-row callbacks over result sets that grow with load, and membership on the request path. Each region is a HYPOTHESIS about where wall-clock time most plausibly concentrates, not a measured fact. The two multiplied dimensions are FREQUENCY (cheap things on every request that add up) and UNIT COST (individually heavy operations).

### Likely time-concentration regions

- **DISTINCT-count-over-subquery (`contract_service.py:131-135`)** — basis: the count is built as `select(query.subquery().c.contract_id).distinct()` wrapped in `select(func.count()).select_from(...)`. When `needs_item_join` is true (search / type_ids / is_bpc / runs / ship_name sort, `:69-77`), the inner query is `Contract OUTER JOIN ContractItem`, so the count is a DISTINCT over a row-multiplied join across potentially tens of thousands of contracts × their items. UNIT COST is high and grows with region size; the DISTINCT forces a hash/sort-aggregate over the full filtered+joined set, not just the returned page. This runs on every request before any page is returned. — confidence: High — also worth the `data-access` and `algorithmic` lanes' attention.

- **Data query: row-multiplying join + `selectinload(items)` (`contract_service.py:167-177`)** — basis: the data query reuses the same `query` object (with the join when present), applies `order_by/offset/limit`, then `selectinload(Contract.items)`. Two cost components: (1) when joined, the base query emits duplicate Contract rows that `.scalars().unique().all()` (`:177`) deduplicates in Python — work proportional to rows-after-join, not to `size`; (2) `selectinload` issues a second round-trip (`SELECT ... FROM contract_items WHERE contract_id IN (...)`) fetching every item for the page's contracts. For a ship-heavy page this is the bulk of bytes transferred and ORM hydration. UNIT COST scales with items-per-contract, independent of page size. — confidence: High — also worth the `data-access` lane's attention.

- **Per-row Pydantic serialization of the page (`contract_service.py:186`)** — basis: `[ContractSchema.model_validate(c) for c in contracts]` runs Pydantic v2 `from_attributes` validation once per contract, and each `ContractSchema` nests `items: List[ContractItemSchema]` (`schemas/contracts.py:47`), so validation also walks every item per contract. FREQUENCY × UNIT COST: bounded by `size` (≤100) contracts but multiplied by items-per-contract; the nested-model validation plus the FastAPI `response_model=PaginatedResponse[ContractSchema]` re-serialization on return (`api/contracts.py:25`) means the page is effectively validated/serialized on the response path. Pydantic v2 core is Rust-fast per field, but the field count × item count × two passes (service `model_validate` + framework response serialization) makes this the largest pure-CPU region on the request path. — confidence: Medium — also worth the `serialization` lane's attention.

- **Two sequential DB round-trips per request (`contract_service.py:134` then `:175`)** — basis: count query is awaited to completion, then the data query is awaited; they are independent but run sequentially, plus `selectinload` adds a third round-trip. On an IO-bound async service, latency is dominated by the sum of these awaits. They could be concurrent (`asyncio.gather`) since neither depends on the other's result (the `total == 0` early-out at `:137` is the only coupling). FREQUENCY: every request pays the serialized latency. — confidence: Medium — also worth the `concurrency` lane's attention.

- **Structured logging per request, two-to-three calls (`contract_service.py:49-61`, `:139-151`/`:190-206`)** — basis: `logger.info("Starting contract search", ...)` fires on entry building a dict, then `log_key_event(...)` fires on completion building another dict that re-lists all filter fields. Both eagerly construct the `search_terms` dict and the structlog processor chain (`logging.py:58-78`) runs merge_contextvars → add_log_level → add_logger_name → ISO TimeStamper → StackInfoRenderer → format_exc_info → JSONRenderer on every record. FREQUENCY is per-request and unconditional (no `isEnabledFor` guard); UNIT COST is the JSON-render of a modest dict. Individually small, but it is fixed overhead on every request regardless of result size. — confidence: Medium — map-only.

- **Middleware stack per request: RequestID + Prometheus instrumentation (`main.py:100-113`, `logging.py:33-47`)** — basis: every request passes through `RequestIDMiddleware.dispatch` (uuid4 generation, `clear_contextvars` + `bind_contextvars`) and the `prometheus_fastapi_instrumentator` middleware (with `should_instrument_requests_inprogress=True` and `inprogress_labels=True`, which updates in-progress gauges with labels on entry/exit). Both are `BaseHTTPMiddleware`-style wrappers, which add an `await call_next` ASGI hop and per-request Python overhead. FREQUENCY: every request, fixed cost; does not scale with result size. The `inprogress_labels=True` label cardinality work is the heavier half. — confidence: Low — map-only.

- **`ContractFilters` dependency parsing/validation (`api/contracts.py:28`, `schemas/contracts.py:68-135`)** — basis: `filters: ContractFilters = Depends(ContractFilters)` makes FastAPI parse ~24 query parameters and run Pydantic validation (min_length, ge bounds, enum coercion for sort_by/sort_direction) on every request. FREQUENCY per request, fixed small cost independent of result size. — confidence: Low — map-only.

- **Per-request session lifecycle + `commit()` on a read path (`db.py:44-57`)** — basis: `get_db` opens an `AsyncSessionLocal()`, yields, then `await session.commit()` and `await session.close()` on every request — including this read-only path. The commit on a read issues a transaction round-trip with nothing to flush (`autoflush=False`, `expire_on_commit=False`). Small fixed per-request overhead, mostly an extra network round-trip to the DB. — confidence: Low — map-only.

### Notes for architecture

- The count and data queries share the same row-multiplying joined `query` object. Counting distinct `contract_id` over an outer join and then deduplicating Contract objects in Python (`.unique()`) are two symptoms of the same structural choice: filtering on a one-to-many relation via JOIN. An `EXISTS`/correlated-subquery formulation for item-based filters, or a `WHERE contract_id IN (SELECT ...)` semi-join, would let the count run over `contracts` alone (no row multiplication, no DISTINCT-over-join) and let the data query avoid emitting duplicate parent rows — collapsing the two heaviest regions at once. This is a design observation, not a defect claim.

- The count and data queries are independent and currently sequential (`:134`, `:175`); structurally they admit `asyncio.gather`, with the `total == 0` early-out folded in afterward. Whether the latency saving is worth the extra concurrent DB connection per request is a load-shape question for the architecture, not a given.

- Sorting by `ship_name` (`SORT_MAP` → `ContractItem.type_name`, `:31`) over an outer join with a one-to-many can produce nondeterministic page contents (a contract with N items has N candidate sort values) in addition to the row-multiplication cost — a correctness-adjacent note that also bears on why that sort path is the most expensive query shape.

- There is a Redis/Valkey cache layer wired into the app (`core/cache.py`, `core/dependencies.py:get_cache`) but the `GET /contracts/` read path does not touch it — the list endpoint recomputes count + data + serialization from Postgres on every request. Whether this list is cacheable (even briefly) given the background aggregation refresh cadence is an architecture question; if it is, caching would short-circuit nearly every region above.

- Both observability middlewares use Starlette's `BaseHTTPMiddleware`, which wraps the response in an extra async generator and is known to add per-request overhead versus pure-ASGI middleware. With only two of them the cost is modest, but it is fixed per request and worth awareness if the stack grows.
