# Execution Cost Map — Frontend (contracts slice)

> Architectural awareness, NOT an optimization to-do list. NOTE: this SPA is largely latent (`app.routes.ts` is `routes = []`), so most regions below are hypothetical-once-wired. The only region executed today is the bootstrap path through `main.ts` → `appConfig`.
>
> Lane: Execution Cost Map (superpowers-plus/performance-audit). STATIC-ONLY, DESCRIPTIVE. Stack: Angular 20 (zoneless + signals), RxJS 7.8, TS 5.8.

## Reachability summary
- **Reachable now:** `main.ts` bootstrap → `appConfig` providers → `AppComponent` renders a bare `<router-outlet>` (`app.component.html:1`) into an empty route table. No feature code runs; no HTTP fires; pipes are never instantiated.
- **Latent-once-wired:** Everything in `features/contracts/*` and `shared/pipes/*`. There is no contract-list component or template in the tree yet (only services, pipes, a resolver, and models exist), so the search pipeline, the per-row pipes, and `@for` list rendering have **no current call site**. The map reasons about where time WOULD concentrate when a list component is added and the resolver/route are wired.
- Note: two parallel, non-wired API surfaces coexist — `ContractSearch` (signal+RxJS, `contract-search.ts`) and `ContractApi` (`contract.api.ts`) — plus duplicate model files (`contract.models.ts` vs `contract.model.ts`). Only `ContractSearch` is referenced by the resolver.

### Likely time-concentration regions

- **App bootstrap + provider graph** — basis: `bootstrapApplication(AppComponent, appConfig)` (`main.ts:17`) instantiates the zoneless scheduler (`provideZonelessChangeDetection`, `app.config.ts:4`), the router with `withViewTransitions()` + `withComponentInputBinding()` (`app.config.ts:15`), the fetch-backed HttpClient (`app.config.ts:16`), and the global error listeners (`app.config.ts:13`). This is one-shot, fixed-size work with an empty route table, so the cost is small and bounded — frequency = once, unit cost = low. — confidence: High — reachable-now — map-only.

- **`ContractSearch` constructor pipeline setup** — basis: the entire RxJS chain is built once in the constructor (`contract-search.ts:55-95`) at first injection. Because the service is `providedIn: 'root'` and lazily injected, this only runs when something first injects it (today: nothing; once wired: the resolver at `contract-filter.resolver.ts:15`). `startWith(undefined)` (`contract-search.ts:58`) means the subscription fires one initial fetch immediately on construction. Frequency = once per service lifetime; unit cost = low (operator wiring + one emission). — confidence: High — latent-once-wired — map-only.

- **Search pipeline per-keystroke path: debounce → map → distinctUntilChanged(JSON.stringify) → switchMap → HTTP** — basis: every `updateFilters()` call pushes `filterTrigger$.next()` (`contract-search.ts:104`). The hot operator is `distinctUntilChanged((prev, curr) => JSON.stringify(prev) === JSON.stringify(curr))` (`contract-search.ts:61`): it serializes the full `ContractSearchFilters` object twice per emission. The 300ms `debounceTime` (`contract-search.ts:59`) collapses keystroke bursts to ~1 evaluation per quiet-period, so frequency is gated low and the filter object is tiny (≤6 scalar fields, `contract.models.ts:57-64`). `switchMap` (`contract-search.ts:63`) correctly cancels in-flight HTTP on a new filter, so the dominant unit cost here is network latency, not CPU. Where CPU time concentrates is the per-emission `JSON.stringify` ×2 and the `HttpParams` rebuild (immutable `.set()` chain, `contract-search.ts:64-79`) — both trivial at this object size. Frequency = per debounced filter change; unit cost = low. — confidence: Medium — latent-once-wired — map-only.

- **State propagation via signals/computed on each HTTP settle** — basis: the subscription does `#state.update(...)` on success (`contract-search.ts:92`) and on the loading/error transitions (`contract-search.ts:62,84`). Each update notifies three `computed` projections — `loading`, `error`, `data` (`contract-search.ts:44-46`) — which are pull-based and only recompute when read by a template binding. Under zoneless CD, each settle schedules exactly one targeted re-render of the bindings that read those signals, not a tree walk. So propagation cost scales with *number of bindings reading the signal*, not component count. Frequency = a few transitions per fetch (loading→data); unit cost = low. — confidence: Medium — latent-once-wired — also worth the change-detection / rendering lane's attention (the `data()` signal is the fan-out point into the list).

- **Per-row pipe evaluation across the rendered list (`isk`, `timeLeft`)** — basis: this is the structurally hottest region once a list template exists. Both pipes are **pure standalone pipes** (`isk.ts:19-22`, `time-left.ts:3-6`), so Angular memoizes by input reference and re-invokes `transform` only when a row's bound value changes — not every CD cycle. At ~100 contracts/page (`size` defaults to 20 but is URL-overridable, `contract.models.ts:58` / `contract-filter.resolver.ts:19`) with, say, price + reward + expiry per row, that is on the order of a few hundred `transform` calls on the *first* render of a result page, then near-zero on subsequent CD passes thanks to purity. Per-call unit cost is small but non-trivial: `Isk.transform` does `Math.abs`, `toFixed`, `parseFloat`, or `Number.prototype.toLocaleString` (`isk.ts:42-57`) — `toLocaleString` is the most expensive branch (Intl formatting) and fires for sub-million values. `TimeLeft.transform` allocates two `Date` objects and does integer arithmetic per call (`time-left.ts:13-37`) — cheap, but note it is a **snapshot at render time**: it computes "time left" once and, being pure with a stable string input, will not tick down without an external refresh trigger. Frequency = O(rows × pipe-bound-fields) per page render; unit cost = low-each, additive across the list. — confidence: High — latent-once-wired — also worth the change-detection / rendering lane's attention (list rendering is the dominant per-page cost; `@for` `track` choice and any future virtual scroll live there).

- **List DOM construction / `@for` rebuild on page or filter change** — basis: there is no list template today, so this is purely prospective. Each new `data()` emission replaces the `items` array reference (`contract-search.ts:92` sets `data` to the fresh response), which means a future `@for` over `data().items` rebuilds or diffs ~up-to-100 rows per page navigation or filter change. Angular 20's `@for` mandates `track`, so the cost depends on whether the eventual template tracks `contract_id` (stable identity → minimal DOM churn) vs `$index` (full row-content rebind on reorder/sort). Sorting is a first-class filter (`sort_by`/`sort_order`, `contract.models.ts:62-63`), so reorders are expected and identity tracking matters. Frequency = per page/filter/sort change; unit cost = medium (DOM diff over the page). — confidence: Medium — latent-once-wired — also worth the change-detection / rendering lane's attention.

### Notes for architecture
- The signal + zoneless design means the usual Angular "tree-wide CheckAlways" cost center does **not** apply here: re-render scope is gated to signal-reading bindings, so the per-page list render (rows × pure pipes) is the natural concentration point rather than change-detection breadth.
- Both list pipes are pure, which is the favorable case — they will not re-run every CD cycle, only on input-reference change. The main per-page cost is therefore the *first* render of a result set, not steady-state.
- `timeLeft` produces a render-time snapshot (`time-left.ts:13-15` reads `new Date()` inside `transform`). Because the pipe is pure, the displayed value is frozen until the bound input changes; a live countdown would require an external tick source feeding the binding — a correctness/UX consideration, flagged here only because it shapes how often the pipe would re-execute if a ticker is later added (that would convert a one-shot per-row cost into a recurring O(rows) cost per tick).
- `distinctUntilChanged` uses `JSON.stringify` equality (`contract-search.ts:61`). Cost is negligible at the current ≤6-field filter shape; the only reason it appears on the map is that it runs on the per-emission path. Key-ordering of `JSON.stringify` is stable for these object literals, so it functions as intended.
- Two unwired API services (`ContractSearch`, `ContractApi`) and two model files duplicate intent. This is a structural/maintenance note, not a runtime cost — neither executes today, and only `ContractSearch` is on the resolver path.
- `size` is URL-driven and unbounded by the resolver (`contract-filter.resolver.ts:19` accepts any parsed integer). The "~100 rows/page" assumption is the analysis ceiling given for this map; a larger `size` would scale the per-row pipe and `@for` regions linearly and would push CDK virtual scroll into scope for the rendering lane.
- HTTP uses `withFetch()` (`app.config.ts:16`); no `withHttpTransferCache()` / SSR hydration is configured, but there is no SSR in this slice, so the double-fetch concern from the Angular lens does not apply.
