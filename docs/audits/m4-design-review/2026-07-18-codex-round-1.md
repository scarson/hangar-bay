# M4 Design Spec — Codex Adversarial Review (Round 1, 2026-07-18)

ABOUTME: Verbatim cross-model (Codex CLI, high reasoning) adversarial review of the M4 production-readiness design spec, run pre-merge on PR #44.
ABOUTME: All 6 P1 + 8 P2 findings were verified against the repo (all held) and applied to the spec in the same PR; the spec's Appendix A records the per-finding disposition.

DONE — findings:

- **[P1] Render’s database URL is incompatible with the async engine as specified.** The blueprint wiring passes Render’s ordinary `postgresql://` connection string directly into `DATABASE_URL`, while both [db.py](/Users/sam/Code/hangar-bay/.claude/worktrees/hangar-bay-frontend-rebuild-2e4fe7/app/backend/src/fastapi_app/db.py:11) and Alembic use `create_async_engine`, which requires an async driver URL such as `postgresql+asyncpg://`. Following [the topology](/Users/sam/Code/hangar-bay/.claude/worktrees/hangar-bay-frontend-rebuild-2e4fe7/docs/superpowers/specs/2026-07-18-m4-production-readiness-design.md:129) would make migrations or application startup fail. The design must bind a URL-normalization strategy and test it against the actual Render reference.

- **[P1] The two-stage baseline plan does not define an immutable, executable migration history.** [Stage 1](/Users/sam/Code/hangar-bay/.claude/worktrees/hangar-bay-frontend-rebuild-2e4fe7/docs/superpowers/specs/2026-07-18-m4-production-readiness-design.md:142) creates a baseline before M3, while [Stage 2](/Users/sam/Code/hangar-bay/.claude/worktrees/hangar-bay-frontend-rebuild-2e4fe7/docs/superpowers/specs/2026-07-18-m4-production-readiness-design.md:143) leaves “regenerate the baseline” and “stack a second revision” as alternatives. Rewriting an applied/stamped baseline can leave databases at that revision without M3’s tables; generating against the current `create_all`-populated dev DB would also produce an empty autogeneration. Decide either to postpone the root baseline until after M3, or retain the Stage-1 root unchanged and add an M3 revision, and require generation/testing against a blank disposable database.

- **[P1] The deploy workflow is not actually gated on the repository’s full CI.** Both workflows trigger independently on a push to `main`, so CI and deploy will run concurrently. The proposed [verify job](/Users/sam/Code/hangar-bay/.claude/worktrees/hangar-bay-frontend-rebuild-2e4fe7/docs/superpowers/specs/2026-07-18-m4-production-readiness-design.md:186) also omits the fixture Playwright lane and `npm run build`. A release can therefore deploy while the canonical CI workflow is still running or ultimately fails. CD should consume a successful CI `workflow_run`, call a reusable full-gate workflow, or reproduce every required gate before invoking Render.

- **[P1] Deploy hooks are treated as synchronous, SHA-pinned deployments without defining either guarantee.** [The workflow](/Users/sam/Code/hangar-bay/.claude/worktrees/hangar-bay-frontend-rebuild-2e4fe7/docs/superpowers/specs/2026-07-18-m4-production-readiness-design.md:186) POSTs two hooks and immediately starts smoke checks; an already-healthy old deployment can satisfy them while the requested build or migration later fails. The claimed previous-SHA rollback is likewise undefined. Bind the requested ref to `github.sha`/a workflow input, wait for each resulting deploy to finish successfully, preserve backend-before-frontend ordering, and verify a deployed release identifier before smoke tests pass.

- **[P1] The smoke gate cannot target production as written.** Only `/api/v1/*` is proxied by the public static-site origin, so polling public `/ready` would hit the SPA fallback rather than the bare backend route; it must use `/api/v1/ready` or the backend service URL. In addition, the “existing” live-smoke project is hardcoded to `https://localhost:5173` and always starts `npm run dev` in [playwright.config.ts](/Users/sam/Code/hangar-bay/.claude/worktrees/hangar-bay-frontend-rebuild-2e4fe7/app/frontend/web/playwright.config.ts:20). The [CD design](/Users/sam/Code/hangar-bay/.claude/worktrees/hangar-bay-frontend-rebuild-2e4fe7/docs/superpowers/specs/2026-07-18-m4-production-readiness-design.md:186) needs a production-base-URL project that disables `webServer`.

- **[P1] Existing production code logs part of the database credential, but the secret-hygiene scope does not remove it.** [background_aggregation.py](/Users/sam/Code/hangar-bay/.claude/worktrees/hangar-bay-frontend-rebuild-2e4fe7/app/backend/src/fastapi_app/services/background_aggregation.py:150) logs the first 30 characters of `DATABASE_URL`, which can include the username and password. The [secret-handling decision](/Users/sam/Code/hangar-bay/.claude/worktrees/hangar-bay-frontend-rebuild-2e4fe7/docs/superpowers/specs/2026-07-18-m4-production-readiness-design.md:174) covers argv, git, and chat but misses this direct production disclosure. Removing/redacting that log must be an M4 prerequisite.

- **[P2] The declared “full prod env inventory” is incomplete.** [The table](/Users/sam/Code/hangar-bay/.claude/worktrees/hangar-bay-frontend-rebuild-2e4fe7/docs/superpowers/specs/2026-07-18-m4-production-readiness-design.md:154) omits the new `METRICS_TOKEN` and all five M3 Settings fields. `METRICS_TOKEN` appears only later and in the Sam-blocked list; a plan derived from the purported authoritative checklist could leave `/metrics` open. Add every post-M3 field and its Render source to the table.

- **[P2] Freshness “outcome” and alert semantics remain undefined.** [The design](/Users/sam/Code/hangar-bay/.claude/worktrees/hangar-bay-frontend-rebuild-2e4fe7/docs/superpowers/specs/2026-07-18-m4-production-readiness-design.md:198) promises `last_success_at + outcome`, but `/ready` exposes only age and the named metric exposes only a success timestamp. Current aggregation catches per-region errors, treats partial data as a committed run, and swallows top-level failures. Define success/partial/failure, when the timestamp advances, the Valkey schema/TTL, reset behavior, stale threshold, and what actually scrapes and alerts on the metric.

- **[P2] Persistence-off Valkey is inaccurately treated as a purely loss-tolerable cache.** The selected instance stores sessions, scheduler state, and concurrency locks as well as disposable cache entries, as shown in [the topology](/Users/sam/Code/hangar-bay/.claude/worktrees/hangar-bay-frontend-rebuild-2e4fe7/docs/superpowers/specs/2026-07-18-m4-production-readiness-design.md:126). Eviction or restart can log out every user and remove operational coordination state. The design should explicitly accept those consequences and bind an eviction policy/capacity alert, or choose persistence/separation/a paid tier before production.

- **[P2] Pool tuning only addresses the request engine, not the ingestion engine.** [Stage 2](/Users/sam/Code/hangar-bay/.claude/worktrees/hangar-bay-frontend-rebuild-2e4fe7/docs/superpowers/specs/2026-07-18-m4-production-readiness-design.md:148) tunes `db.py`, but every aggregation run creates another default engine in [background_aggregation.py](/Users/sam/Code/hangar-bay/.claude/worktrees/hangar-bay-frontend-rebuild-2e4fe7/app/backend/src/fastapi_app/services/background_aggregation.py:151). The implementation plan must either reuse the injected session factory or apply the production connection policy to both engines.

- **[P2] The spec incorrectly assigns the future scope-bearing backlog to M3.** [The M4 statement](/Users/sam/Code/hangar-bay/.claude/worktrees/hangar-bay-frontend-rebuild-2e4fe7/docs/superpowers/specs/2026-07-18-m4-production-readiness-design.md:209) says scope expansion is M3’s thread, but the authoritative [M3 design](/Users/sam/Code/hangar-bay/.claude/worktrees/hangar-bay-frontend-rebuild-2e4fe7/docs/superpowers/specs/2026-07-17-m3-account-features-design.md:15) explicitly makes M3 zero-scope and defers the first ESI scope to an unnamed future milestone. Correct this so the backlog is not mistakenly considered owned or completed by M3.

- **[P2] The Phase-1 collision claim leaves one known shared file unsequenced.** M4 says [.env.example gains production documentation](/Users/sam/Code/hangar-bay/.claude/worktrees/hangar-bay-frontend-rebuild-2e4fe7/docs/superpowers/specs/2026-07-18-m4-production-readiness-design.md:174), while the M3 plan modifies that same file multiple times. It is absent from the execution table, which then claims Phase 1 touches no M3 file. Explicitly assign this edit to post-M3 Phase 3 or acknowledge and plan the rebase.

- **[P2] Required edge hardening from the cited recon disappears from the design.** The recon identifies HSTS, compression, immutable caching for hashed assets, and no-cache handling for `index.html` as production gaps, but none is bound in the Render topology or execution phases. Decide whether Render supplies each requirement or add explicit response-header/cache rules; otherwise the implementation plan has no owner for requirements already established by the security and performance specifications.

- **[P2] The proposed production SSO warning still misses common fatal configurations.** Merely removing the environment guard from `warn_if_sso_unconfigured` leaves the underlying check limited to client ID and cipher keys; [require_sso_configured](/Users/sam/Code/hangar-bay/.claude/worktrees/hangar-bay-frontend-rebuild-2e4fe7/app/backend/src/fastapi_app/api/auth.py:36) likewise ignores an empty `ESI_CLIENT_SECRET` and localhost callback/origin defaults. Expand the production diagnostic to validate all required SSO fields and reject or clearly warn on localhost production URLs.
