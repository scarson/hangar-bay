## P1 findings (block or corrupt execution)

### [P1] Task 3.4 Step 2 (plan line ~902) — wrong import: `get_db` is not in `core/dependencies`
`api/ops.py` imports `from ..core.dependencies import get_cache, get_db`. Verified: `get_db` is defined **only** in `app/backend/src/fastapi_app/db.py:30`; `core/dependencies.py` exports `get_cache`, `get_http_client`, `get_esi_client` — no `get_db`. As written, `ops.py` raises `ImportError` at load and the app won't start. Compounding: the house test fixtures (`conftest.py`) override `fastapi_app.db.get_db`, so even if the import were patched to some other symbol, the `/ready` DB override wouldn't apply.
**Fix:** `from ..db import get_db` and `from ..core.dependencies import get_cache`. Also correct the Step-2 note "Adapt `get_db`/`get_cache` names to the merged `core/dependencies.py` exports" — `get_db` lives in `db.py`, not `dependencies.py` (M3 does not move it: M3 touches `dependencies.py` only once and does not touch `db.py`).

### [P1] Task 3.3 (plan lines ~799–855) — freshness outcome mislabels 304 (not-modified) runs as "failure"
`run_aggregation` catches `ESINotModifiedError` in a **separate** `except` (background_aggregation.py:172–173) that does not hit the `logger.error("Failed to fetch...")` site (:174–175). The plan increments `failed` only at the error site and leaves "ok" as successful fetches. Once ETags populate (every steady-state run after the first), a region returns 304 → counted as neither ok nor failed. With the single configured region `[10000002]`, a healthy repeat run yields `ok=0, failed=0` → `outcome="success" if failed==0 and ok>0` is **False** → `"failure"`, and `last_success_at` is **not** advanced. `/ready` then reports `last_ingest_outcome="failure"` and `data_stale=true` on a perfectly healthy deploy. The three planned tests (all-succeed / one-fails / all-raise) never exercise the all-304 case, so the bug ships green.
**Fix:** treat a not-modified region as `ok` (data confirmed current — the success timestamp should advance), or redefine `outcome` in terms of `regions_attempted` vs `regions_failed` rather than requiring `ok>0`; add an explicit all-not-modified test.

## P2 findings (advisory)

### [P2] Task 3.5 Step 2 (plan lines ~977–988) — `/metrics` snippet uses `HTTPException` without importing it
`main.py` imports `FastAPI, Depends, APIRouter` and `Request` only — not `HTTPException`. The GREEN snippet's `raise HTTPException(status_code=401, ...)` will `NameError`. Add `from fastapi import HTTPException` to the snippet. (`PlainTextResponse` and the `prometheus_client` imports are already in the snippet; `JSONResponse` is imported but `PlainTextResponse` is not — the snippet adds it, good.)

### [P2] Task 3.3 Step 2 (plan lines ~832–855) — `_record_run_outcome` redis client is unspecified, and "inside the lock" is inaccurate
`_record_run_outcome(self, redis_client, ...)` needs a redis client, but the lock's client is created and closed **inside** `_concurrency_lock` and never exposed to `run_aggregation`. The plan doesn't say where the client comes from (mirror `aioredis.from_url(str(self.settings.CACHE_URL))`). Also the plan says it's "called from the run's success and failure exits (inside the lock)" — the generic `except Exception` failure exit (:198) runs **after** the lock's `finally` released it, so that path is not inside the lock. Specify the client acquisition, correct the description, and ensure the write path is the same one the aggregation tests patch (the FakeRedis seam) or the freshness assertions can't observe the write.

### [P2] Task 3.9 Step 4 (plan lines ~1074–1094) — `blank_migrated_sync_connection` cites a non-existent "house idiom" for CREATE DATABASE
The fixture comment says to create/drop the throwaway DB "via conftest's existing DB-provisioning idioms" / "house idiom." Verified: `conftest.py` has **no** CREATE DATABASE idiom — its `db_session` fixture does `drop_all`/`create_all` against an already-existing DB; the test database is created by the **CI workflow's** `psql ... CREATE DATABASE` step, not by conftest. A cold subagent will hunt for code that isn't there. Spell out the real requirement: open an autocommit `psycopg2` connection to the `postgres` maintenance DB, `CREATE DATABASE m4_equiv_check`, then `DROP` it in teardown. (The `.replace("+asyncpg","+psycopg2")` and `parents[2]/alembic.ini` path are both correct; psycopg2-binary is installed.)

### [P2] Task 3.9 Steps 1 & 3 (plan lines ~1044, ~1052) — speculative M3 schema names don't match M3
The plan guesses M3 model modules `saved_search`/`watchlist`/`notification` and a `notifications` partial-unique-index table. The M3 plan actually adds `models/account.py` (plus `watchlist_alerts_enabled` on `user.py`). The authoritative instructions ("import exactly what `models/__init__.py` exports"; hand-review the generated revision) are correct and save the executor, but the guessed names should be marked clearly illustrative so a cold subagent doesn't import nonexistent modules or search for a `notifications` table that may be named differently.

### [P2] Task 1.2 Step 1/2 (plan lines ~266–358) — call out the two Render blueprint keys most likely renamed
`autoDeployTrigger: off` and the Key-Value `fromService: {type: keyvalue, property: connectionString}` are the keys Render has historically renamed (`autoDeploy` ↔ `autoDeployTrigger`). Step 2 already mandates validating against the live Blueprint spec — name these two explicitly so validation isn't perfunctory. (Not a hard error since the plan self-guards with the spike dry-run.)

### [P2] Task 3.4 Step 1 (plan lines ~866–887) — `/ready` tests must also override `get_cache`
The house `client`/`test_app` conftest fixtures wire only `get_db`; they do **not** set `app.state.redis` (only `auth_client` does). So `get_cache` will raise 503 unless the `/ready` tests add `dependency_overrides[get_cache]` returning a fake redis. The plan says "fake redis" but doesn't note that the standard `client` fixture leaves redis unwired — a cold subagent using `client` as-is will get spurious 503s.
